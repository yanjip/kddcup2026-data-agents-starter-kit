# 失败模式 — 禁止重复

## 1. 必须先将问题关键词映射到 knowledge.md
- 写任何查询前，先把问题里的每个关键词映射到 knowledge.md 的精确定义
- 尤其注意：Metric Definitions、Ambiguity Resolution、Exemplar Use Cases 三个区块
- 如果 knowledge.md 里没有你要的定义，**停止反复搜索**，直接基于实际数据推断

## 2. 输出列必须严格匹配问题要求
- 只返回问题明确要求的列，**不要多加中间计算列**
- "List all X" = 只返回 X 那一列，不要返回整行
- 多个指标必须横向输出（多列在一行），禁止纵向 pivot（多行）

## 3. 工具失败必须换策略，不能重试
- **同一代码或同一工具失败 2 次后，第 3 次必须改用完全不同的方法**
- `execute_python` JSON 解析报错 → 检查代码中是否有嵌套双引号，改用单引号或三引号
- `execute_python` 反复报错 → 立刻切换到 `read_doc` / `read_csv` / SQL 工具
- 不要陷入"再试一次可能就好了"的思维陷阱

## 4. 每个问题拆成三步自检
- Filter：我要按什么条件过滤？
- Compute：我要算什么？（比率？总和？平均？计数？）
- Return：我要返回什么列？
- 每步完成后问自己：这和问题的字面意思一致吗？

## 5. 对零值和异常值保持警惕
- 求最小/最大值得到 0 时，先确认 0 是真实记录还是占位符/缺省值
- 返回结果行数明显偏多或偏少时，重新检查过滤条件
- 返回结果中空值超过一半时，重新检查数据源和 join 条件

## 6. Python 代码书写规范
- Python 代码中的字符串优先用单引号 `'` 或三引号 `"""`，避免在 JSON 中出现嵌套双引号
- 不要用 `subprocess.run` 调 grep 来解析 JSON 文件，直接用 Python 内置的 `json` 模块

## 7. 大文档处理原则
- 不要反复用 `read_doc` + offset 去"翻页"找信息
- 先用一次 `execute_python` 做关键词扫描，找不到就基于已有信息做合理推断
- 不要在单个大文档上浪费超过 3 个 step
